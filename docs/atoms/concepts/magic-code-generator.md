## Definition

A **magic code generator** is a cell-magic whose execution produces valid magic syntax — text that the parser dispatches as **new cells** or as additional operations. Operator-designated and operator-trusted, generators are an explicit exception to the [emission ban](../discipline/magic-injection-defense.md) that S5.0.1 places on agent and tool output: agents must NEVER emit dispatchable magic syntax; generators legitimately do, because their entire purpose is to produce more magic. Generators are the recursive case of the [certified magic emitter](../discipline/certified-magic-emitter.md) standard — they satisfy clause 1 (operator-rooted intent) by virtue of the operator typing the generator-magic cell, and clauses 2–5 by routing through Cell Manager with `generated_by` provenance.

The contrast holds:

| Source | Output | Treatment |
|---|---|---|
| Operator typing (cell editor) | Magic syntax | Dispatched (always) |
| Magic code generator (operator-designated) | Magic syntax | Dispatched (with provenance) |
| Agent stdout / tool result | Anything resembling magic syntax | Layer 1: contamination flag. Layer 2 (hash mode): emission ban — escape leading `@` |

Generators are the third path. Without them, repetitive operator workflows ("spawn 5 agents from a list," "expand a template," "import cells from another file") have no expressive primitive — operator either types each cell by hand or breaks the visible-tile discipline by hand-editing kernel state.

## Schema additions (when the slice ships)

```jsonc
metadata.rts.config.magic_code_generators: [<magic_name>, ...]
// V1 built-ins: ["template", "expand", "import"]
// V2+: operator-registered custom generators append here

metadata.rts.cells[<id>] = {
  text, outputs, bound_agent_id,                     // existing
  contaminated, contamination_log,                    // from S5.0.1
  generated_by: <cell_id> | null,                    // NEW — provenance
  generated_at: <ISO8601> | null
}
```

Every cell created by a generator carries a `generated_by` pointer back to the originating generator cell. Provenance chains are valid: a generator that itself was generated by another generator preserves the originating cell. Operator can trace any cell back to the operator-typed root.

## Trust model — three rules

1. **Operator-designated only.** The generator registry is operator-controlled. Built-in V1 generators ship with the kernel; V2+ allows operator-registered custom generators (Python functions). Agents cannot register or invoke generators, and an agent emitting `@@template foo` in its output is just text — Layer 1 contamination flag fires; Layer 2 (in hash mode) escapes the line. Only operator-typed generator-magic cells ever execute.

2. **Generators run in operator privilege.** When the generator handler runs, the kernel grants it access to the operator's pin (when hash mode is on), so its emitted magic lines auto-include valid HMACs per [discipline/magic-injection-defense](../discipline/magic-injection-defense.md). This is the **sole legitimate path that produces valid hashed-magic syntax through any output mechanism**. Agent stdout never produces it; the emission-ban sanitizer never sees it (generators write structurally via [cell-manager-owns-structure](../discipline/cell-manager-owns-structure.md), not through the output stream).

3. **Generated cells track provenance.** New cells inserted by a generator carry `generated_by` + `generated_at`. The provenance is part of the canonical cell metadata and survives `.llmnb` round-trip. UI surfaces it as a small chip on the generated cell ("from `@@template setup_pipeline` in c_3").

## V1 built-in generators

| Generator | Body | Effect |
|---|---|---|
| `@@template <name> [args]` | none, or kwarg overrides | look up `metadata.rts.config.templates[<name>]`; split + insert as new cells after the generator cell |
| `@@expand` | a here-doc of cell magics | parse body as a notebook fragment; insert each `@@break`-bounded cell after the generator cell |
| `@@import <file>` | none | read another `.llmnb` file from workspace; copy cells (with `generated_by` provenance) into the current notebook |

V1 generators are pure templating — no loops, no conditionals, no operator-defined Python. V2+ adds `@@for_each`, `@@if`, `@@compose`.

## Why this is Zachtronics-compliant

The visible-tile discipline ([zachtronics](../discipline/zachtronics.md)) requires that operator-visible structure live in the cell text. Generators preserve this:

- The generator-magic cell is operator-typed and visible.
- The generated cells are placed in the notebook after the generator cell — operator sees them too.
- The provenance chip makes the generator-to-generated relationship inspectable at a glance.
- Templates themselves (`metadata.rts.config.templates`) are operator-defined visible config, not hidden kernel logic.

What a generator is NOT: a "macro that runs hidden code with side effects beyond the visible cell list." Generators only produce cells; they don't mutate kernel state outside that. The cells they produce are first-class, editable, deletable. Operator owns them.

## Anti-shapes ruled out

| Anti-shape | Why forbidden |
|---|---|
| Agent emits `@@template foo` in stdout → kernel runs the template | Layer 1 contamination flag fires; agent-stream prose never executes generators. Only operator-typed generator cells dispatch. |
| Operator-registered generator runs untrusted code from agent input | Generator handlers are kernel-side functions; their inputs are operator-typed args. Agent text never reaches them as code. |
| Generator output bypasses pin verification in hash mode | Generators that produce hashed-magic output MUST compute valid HMACs. Generator handler API requires it; lint enforces. |
| Generated cells without provenance | Rejected — every cell created by a generator MUST carry `generated_by`. |
| Agent triggers a custom generator registration via prose | Forbidden — registration is an operator-only intent, never an agent-emitted side effect. |

## V1 vs V2+

- **V1 (shipped, PLAN-S5.0.2 / submodule `33b5c50`)**: three built-in generators (`@@template`, `@@expand`, `@@import`). Pure templating. No loops. No operator-registered custom handlers. Provenance schema lands. K3H / K3I / K3J registered.
- **V2+**: `@@for_each` / `@@if` / `@@compose` (control-flow generators). Operator-registered custom handlers via a registration intent. Generator chains with multi-step provenance.
- **V3+**: cross-notebook generator pipelines, generator-output diffing for pre-execution review.
